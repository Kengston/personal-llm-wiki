# ADR-0015 — Permission-постура capture-write-path: движок пишет файлы (`acceptEdits`, без shell), коммитит мост

> Кристаллизовано через `/grill-with-docs` (Phase 7, активация capture 7.3). Уточняет постуру движка из [ADR-0007](0007-engine-spawn-and-scheduler.md) до конкретного механизма; **ничего не superseded** — `compiler/rules.md §11` («движок не пишет git») остаётся буквально верен.

**Контекст.** Phase 7 включает **capture** ([SETUP §7.3](../../setup/SETUP.md)): owner шлёт факт/идею в Telegram → движок должен **дописать страницу** в `wiki/` приватного репо. До Phase 7 контракт хранителя подразумевал **ручные, человеко-ревьюабельные коммиты**: `compiler/rules.md §0` требует «маленькие ревьюабельные коммиты — git-diff на каждый ход», а `§11` прямо запрещает движку git/push («нормальный git-flow человека»). Автономный Telegram-capture требует **авто-коммита**, иначе нет «git-diff на ход» и не выполняется DoD 7.3. Это и есть дорого-обратимое решение, ради которого заведён ADR.

Граничные факты, вскрытые при гриллинге:
- **[ADR-0007] уже предписывает** искомую постуру: *«least-privilege-доступ движка к ФС (запись только в приватный репо)… без generic «fetch any URL»/shell-инструмента»*. То есть **запись файлов в приватный репо — не новая привилегия**, а ровно действующий инвариант; чего ADR-0007 не хочет — это **shell** у движка (lethal-trifecta-митигация).
- **`acceptEdits` Claude Code НЕ покрывает git** (проверено на `claude` 2.1.92 + офиц. доки): авто-аппрувит правки файлов и узкий набор fs-команд (`mkdir/touch/rm/rmdir/mv/cp/sed`) в cwd, но `git add/commit` — Bash к защищённому `.git/` → в headless `-p` блокируется (нет аппрувера). Значит «движок правит файлы» и «движок коммитит» — **разные** права.
- **`§2`** требует, чтобы входящий capture-текст прошёл `sanitizeText()` **до** записи; LLM-движку маскирование доверять нельзя (нужен детерминированный write-path-код).

**Решение.**

1. **Движок пишет только файлы, без shell/сети.** Постура `acceptEdits` (Read/Write/Edit + benign fs в cwd); `git`, `curl`, произвольный Bash и сетевые инструменты — недоступны (в headless блокируются). Закреплено **декларативно** в `.claude/settings.json` **приватного** content-репо (cwd движка): `{"permissions": {"defaultMode": "acceptEdits", "deny": ["WebFetch", "WebSearch"]}}`. Авто-грузится для **каждого** `claude -p` в этом каталоге → единый источник постуры для **моста (реактив), sweep (плановое) и ручных прогонов**, ноль дублирования env.
   - *Engine-portable ([ADR-0008]):* эквивалент постуры для отложенного `CodexEngine` — `~/.codex/config.toml` (`approval_policy="never"`, `sandbox_mode="workspace-write"`, `writable_roots=[<content-repo>]`, `network_access=false`), как уже описано в [ADR-0007]/[SETUP Appendix B]. Постура — кросс-движковая; механизм у каждого движка свой.

2. **Коммит per-turn делает МОСТ (доверенный TS-код), не движок.** После каждого хода мост проверяет `git -C <content-repo> status --porcelain`; если рабочее дерево «грязное» → `git add -A && git commit -m "note: …"`. **Query-ходы чисты → без коммита**; capture-ходы грязны → ровно один коммит. Так:
   - `§11` «движок не пишет git» — **буквально соблюдён** (гитует доверенный мост, не LLM);
   - `§0` «git-diff на каждый ход» — **сохранён** (каждый capture = один ревьюабельный коммит);
   - [ADR-0007] «без shell у движка» — **соблюдён** (мост — это код, а не инструмент LLM).

   **Плановый слой симметричен:** правки sweep'а (reminders `status`/`last_fired`, `reminders/log.md`) коммитит доверенный `scheduler/run_sweep.sh` (verb `audit`), а не движок — та же постура (`acceptEdits`, без shell) применяется к `claude -p` дайджеста в том же cwd.

3. **Push остаётся ручным** (`§11` «не пушит»; пуш — только по явному подтверждению владельца). Авто-коммит локальный; ревью — post-hoc через `git log`/`git show` (как и делает тест 7.3).

4. **Capture-текст маскируется мостом `failClosedSanitize()` ДО движка** (инвариант `§2`): owner-сообщение проходит маскер в inbound-пути моста до формирования промпта; сбой маскера → ход отменяется целиком (fail-closed). Облачный движок никогда не видит сырой секрет ([ADR-0009] «privacy движка»). Применяется равномерно к query+capture (для обычного текста — тождество).

**Альтернативы (отвергнуты).**
- **R2 — движку узкий `Bash(git add/commit)` через `--allowedTools`.** Меньше кода, но **даёт LLM shell** — прямо против [ADR-0007] (минимизация blast radius, lethal trifecta: приватные данные + недоверенный capture-текст + действие). Отвергнуто.
- **R3 — без авто-коммита** (ручной/плановый коммит позже). Ломает `§0` «git-diff на ход» и DoD 7.3. Отвергнуто.
- **Механизм — флаг в `engine.ts` (`CLAUDE_EXTRA_ARGS=--permission-mode acceptEdits`).** Не покрывает ручные прогоны, требует дублировать env в публичный (мост) и приватный (sweep) `.env` — легче забыть путь. Слот `extraArgs` сохранён как **override-хук**, но не как носитель постуры.

**Следствия и явное принятие риска.**
- **raw/-иммутабельность в v1 держится контрактом (`§11`) + git-backstop, НЕ permission-правилами.** `acceptEdits` разрешает `rm/mv` в cwd → движок технически может удалить файл в `raw/`. Принятый риск: blast radius = рабочее дерево приватного репо, восстановимо через git; контракт это запрещает, аудит — в diff. Опциональное усиление (per-path `deny`) — отложено, не v1.
- `acceptEdits` авто-аппрувит `mkdir` в cwd — нужно для создания новых поддиректорий (`wiki/ideas/` и т.п.). Всё за пределами cwd, защищённые пути и прочий Bash — блокируются.
- `settings.json` живёт в **приватном** репо (это конфиг-постура, не секрет; коммитится). Публичный фреймворк-портфолио демонстрирует постуру **синтетическим** `wiki-example/.claude/settings.json` + шагом в [SETUP](../../setup/SETUP.md) — без утечки приватных данных ([ADR-0003]).
- Telegram-push (реактив **и** дайджест) делает TS-код моста/scheduler, **не** движок-shell → постура «без shell» исходящий канал не ломает.
- Мост получает новую обязанность: writeback-коммит после хода (детерминированный, идемпотентный по «чистоте» дерева). Sweep продолжает помечать `status`/`last_fired` файловыми правками под той же постурой.
