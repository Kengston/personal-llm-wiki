# ADR-0018 — Финансовый модуль: per-source ingestion, мультивалютный леджер, секреты

> Кристаллизовано через `/grill-with-docs` (2026-06-13). Расширяет [ADR-0003](0003-two-repos-public-private.md) (public/private split), [ADR-0011](0011-relevance-sensitivity-filter.md) (финансы = знание, маскируем карты/IBAN), [ADR-0010](0010-wiki-content-model.md) (контент-модель), [ADR-0002](0002-no-embedder-pure-karpathy.md) (без SQL/вектора); наследует ToS-постуру [ADR-0009](0009-tos-safe-engine-access.md).
>
> Личный per-source маппинг КОНКРЕТНЫХ банков владельца — в ПРИВАТНОМ репо (`llm-wiki-content/docs/finance-assistant-architecture.md`), НЕ здесь: публичный репо не должен раскрывать банковский след владельца. Этот ADR описывает архитектуру обобщённо.

**Контекст.** «Второй мозг» расширяется до персонального финансового ассистента: трекать балансы, доходы, расходы и кредиты по разнородным счетам (банки с API и без, e-wallet, крипто-биржа, зарубежный банк), ставить финансовые цели и проактивно следить за их соблюдением. Жёсткий инвариант: ни токен, ни финансовый PII не должны попасть в публичный репо. Реальность доступа к данным резко разная по источникам — у провайдеров РФ нет открытого API к данным физлица (открытый банкинг ЦБ для розницы — не раньше ~2027), поэтому единого способа сбора не существует.

**Решение.**

- **Per-source ingestion — тиры по реальной доступности данных (white-API-first), а не один универсальный способ:**
  1. **Белое read-only API** — где провайдер реально даёт его физлицу (e-wallet, крипто-биржа, брокерский срез). Приоритет: автономно, дёшево, ToS-чисто, read-only ключ без прав на вывод/торговлю.
  2. **Поток уведомлений** — для банков без API: транзакционные SMS/push читаются ЛОКАЛЬНО (база Сообщений macOS при форварде с iPhone, либо выделенный Android-апплайнс с `NotificationListener` → локальный эндпоинт). Ноль логина в банк, ToS-чисто, near-real-time. Даёт поток операций (расход/доход), но не остаток кредита.
  3. **Браузер по запросу** — Chrome MCP поверх вручную залогиненной сессии для точного баланса и остатка по кредиту (то, чего нет в уведомлениях). On-demand, требует человека для OTP, не автономно; строго read-only, низкая частота.
  4. **Ручной ввод** — статические факты (условия кредита: сумма/ставка/срок) и универсальный фолбэк.

  Режим выбирается ПО ИСТОЧНИКУ исходя из доступности; «белое API сначала», деградация по тирам.

- **Леджер — append-only JSONL** в приватном репо `raw/finance/`: `accounts`, `snapshots` (баланс на момент синка), `transactions`, `credits`, `fx_rates`. Не SQL и не вектор ([ADR-0002](0002-no-embedder-pure-karpathy.md)). Коммитится авто-свипом ТОЛЬКО в приватный репо; в `wiki/finance/` промоутятся лишь огрублённые сводки (числа в прозе вики — корзинами, не точные).

- **Мультивалютность, currency-agnostic:** каждая сумма хранится в НАТИВНОЙ валюте (ISO-4217 + крипто-символы), без захардкоженного списка валют — новые валюты (РФ/крипто/любые) работают без правки кода. `fx_rate` фиксируется на момент синка (pluggable провайдер курса на любую пару) для исторической точности. Роллапы показываются в НАСТРАИВАЕМОМ наборе отображаемых валют (₽/$/₾/…); единой «базовой» валюты НЕТ. Цель деноминирована в своей валюте.

- **Финансовая цель = `type: finance-goal`** (новый тип контент-модели [ADR-0010](0010-wiki-content-model.md); виды: `save` / `spend_cap` / `debt_paydown` / `grow`). Прогресс ВЫЧИСЛЯЕТСЯ детерминированным компаратором из леджера — в отличие от `growth` (качественная, самоотмечаемая цель). Отклонения от цели — в проактивный Telegram-дайджест.

- **Модель секретов (приоритет №1 — ничто не утекает в публичный репо):**
  - Standing read-only токены — только у white-API источников; тиры уведомлений и браузера НЕ держат ни одного standing API-кредла.
  - Хранилище: для MVP — приватный `.env` (gitignored); миграция в **macOS Keychain** на этапе харднинга. В `.env` ПУБЛИЧНОГО репо финансовых секретов быть не должно никогда.
  - **Phase-0 safety gate обязателен ДО первого живого токена** (приземлён и проверен зелёным): правила санитайзера + парность в `lint-public` под банк-токены и крипто-адреса (только assigned-form `KEY=…` + энтропийный детектор; bare fixed-length запрещены — дают массовые ложняки), харднинг `.gitignore` (выписки/сессии браузера/скрины/логи — игнор; `raw/finance/*.jsonl` — трекается), CI-линт публичного репо.

**Альтернативы (отвергнуты).**
- *Единая «базовая» валюта.* Отвергнуто: владелец держит несколько валют — нативное хранение + набор отображаемых, а не схлопывание в одну.
- *Реверс-инжиниринг мобильных API банков («чёрное API»).* Хрупко, нарушает ToS, аппаратная аттестация (Play Integrity) практически непробиваема.
- *Облачный агрегатор (Zenmoney и пр.) как основной источник.* Данные уезжают с машины — против инварианта приватности; допустим лишь как крайний фолбэк для хрупкого коннектора.
- *«Полный автомат везде одним способом».* Миф для тяжёлых банков; реальный ответ — гибрид по источникам.

**Следствия.**
- Новые коннекторы под `src/ingest/finance/` (по тиру на источник); routines `finance_sync` (сбор) + `finance_monitor` (сверка целей, алерты); модуль-компаратор целей; область `wiki/finance/` + страницы `type: finance-goal`.
- Phase-0 gate уже в дереве (`src/ingest/sanitizer.ts`, `src/scheduler/lint-public.ts`, `.gitignore` обоих репо, `.github/workflows/lint-public.yml`) — тесты/линт зелёные.
- Раскатка пофазно: white-API источники первыми (MVP), затем тир уведомлений и браузер по банкам.
- Авто-свип приватного репо коммитит леджер; публичный репо пушится только руками через pre-commit-хук.
