# ADR-0016 — Персона бота: настраиваемый системный промпт реактивного моста (контент приватный)

> Кристаллизовано через `/grill-with-docs` после Phase 7. Уточняет «персону» из [CONTEXT §5](../../CONTEXT.md) / handoff §6; ничего не superseded.

**Контекст.** После Phase 7 (capture/sweep работают) всплыла проблема голоса: `claude -p` отвечает как generic-ассистент, читая контракт хранителя ([compiler/rules.md](../../compiler/rules.md)), но **без персоны**. В живом чате бот называл себя «**движком**» (нарушая [CONTEXT §5](../../CONTEXT.md): «движок» — внутренний термин, «Второй мозг» — система, «бот» — лишь интерфейс), вытаскивал в чат внутрянку (`scheduler`/`launchd`/`bridge`/`raw`/`wiki`) и **врал про runtime** («планировщик не настроен» — он не интроспектит launchd). Нужна персона. Но её **конкретный текст — личные данные** владельца, а публичный репо — фреймворк-продукт «для всех» ([ADR-0003](0003-two-repos-public-private.md)).

**Решение.**

1. **Персона = системный промпт РЕАКТИВНОГО моста**, доставляется через `claude -p --append-system-prompt <persona>` (опция `systemPrompt` на `ClaudeEngine`, [engine.ts](../../src/bridge/engine.ts)). Сообщение владельца идёт **чистым** user-аргументом. **Скоуплено на мост**: плановый sweep строит движок без `systemPrompt` (у проактива свой промпт — [digest.ts](../../src/scheduler/digest.ts)). Engine-portable ([ADR-0008](0008-engine-claude-native.md)): отложенные `GrokEngine`/`CodexEngine` добавят свой маппинг `systemPrompt` → их флаг.

2. **Персона ≠ контракт.** Это слой **моста**, НЕ `CLAUDE.md` (тот — контракт хранителя вики + auto-memory движка). Голос отделён от правил ведения вики.

3. **Регистр-aware** (владелец — и пользователь, и разработчик): по умолчанию голос помощника (прячет внутренние термины, говорит «записал/напомню»); на ПРЯМОЙ вопрос про устройство/архитектуру/Claude Code — отвечает прозрачно. Никогда не зовёт себя «движком» ([CONTEXT §5](../../CONTEXT.md)). Плюс runtime-честность (не утверждать статус планировщика/доставки, которого не видит).

4. **Контент персоны — ЛИЧНЫЙ → приватный репо** ([ADR-0003](0003-two-repos-public-private.md)): реальная персона живёт в `<WIKI_REPO_PATH>/persona.md` (путь — `BRIDGE_PERSONA_FILE`, дефолт), **full-replace** (есть файл → он целиком и есть системный промпт). Публичный фреймворк несёт лишь: настраиваемый слот, **generic `DEFAULT_PERSONA`** в коде (fallback — продукт работает из коробки без личных данных) и **синтетический `persona.example.md`**. Ноль личного в публичном (`lint:public`). Та же связка, что `.env`/`.env.example`, `wiki/`/`wiki-example/`.

**Альтернативы (отвергнуты).**
- **Персона в `CLAUDE.md`.** Смешивает контракт и личность; `CLAUDE.md` — engine-memory, не чистый системный промпт.
- **Персона захардкожена в публичном фреймворке.** Утечка личных данных в публичный репо — нарушение [ADR-0003] (и явное возражение владельца: «это мой личный промпт, не на уровне продукта»). Отвергнуто.
- **Prepend к сообщению владельца** (как было в [ADR-0015]-конверте). Персона сидит в user-турне — слабее, мешается с контентом. Заменено на `--append-system-prompt`.
- **Всегда голос помощника** (без регистра). Владелец не смог бы спросить бота про его же устройство. Отвергнуто в пользу регистр-aware.

**Следствия.**
- Мост грузит персону на старте (`loadPersona`, [prompt.ts](../../src/bridge/prompt.ts)); нет файла/пуст → `DEFAULT_PERSONA`. Sweep не затронут.
- **Full-replace ⇒ персональный `persona.md` обязан нести и роутинг** (query/capture, «git не коммить») — поэтому `persona.example.md` его уже содержит (старт-точка); generic-дефолт — тоже полный (роль+регистр+роутинг).
- Персона — не замена знанию о владельце: факты бот берёт из вики (`profile.md` и др.), персона задаёт лишь голос/границы.
- Отложенные движки при реализации добавляют свой `systemPrompt`-маппинг.
