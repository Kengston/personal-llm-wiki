# ADR-0022 — Ингест knowledge-package нормализует frontmatter под контракт хранителя

**Контекст.** Срез #02 Фазы 1 («Поиск работы») втягивает портативный knowledge-package коллеги (`job-search-knowledge` v0.2.0, источник `nazarov-2024`) в приватный контент-репо. Пакет собран по **ТЗ-форме** своего репо: `sources: -{id, ref, source_date}`, `claims:` — **простые строки** (атомарные утверждения источника), `status: draft|review|accepted`. Живой же **контракт хранителя** ([compiler/rules.md §3a.1](../../compiler/rules.md), [ADR-0019](0019-knowledge-library-external-methodology.md)) ждёт **object-форму**: `sources: -{name, date, freshness, url}` и `claims: -{id, text, confidence, status, sources}` со словарём `status: draft|active|verified|stale|superseded|archived` (§4). Seed-страница `resume-ats.md` уже была в контрактной форме. Простая копия дерева оставила бы методологию в чужой форме: движок маршрутизирует и взвешивает по контракту хранителя, а не по ТЗ-форме пакета. Механической поломки `build`/`lint:public` это не даёт (`build` = `tsc`, `lint:public` = секреты/PII — frontmatter никто не валидирует на этапе сборки), но это **семантический** дрейф контракта.

**Решение.** Ингест — **не чистая копия**: он детерминированно **нормализует frontmatter каждой страницы под контракт хранителя**, с одним осознанным ослаблением контракта. Правила трансформации:

1. **`sources:` ТЗ-форма → object-форма.** `{id, ref, source_date}` → `{name, date, freshness, url}`: `name ← ref` (человекочитаемая атрибуция), `date ← source_date`, `url ←` относительный путь страницы до сырого транскрипта в `raw/` (см. п.5), `freshness ←` **производная от возраста источника** (≤12 мес — `current`, ≤24 — `aging`, >24 — `stale`); для `nazarov-2024` (2024-09 → ингест 2026-06, ≈21 мес) → `aging`. Производная — не выдуманный факт, а документированная эвристика свежести.
2. **`claims:` строка → структура `{id, text, status, sources}`** (БЕЗ `confidence`): `id ← claim_<4hex>` от `sha256(slug + '::' + text)` — детерминирован и стабилен при ре-ингесте (нужен для будущей механики кросс-источниковых противоречий §3a.4); `text ←` исходная строка с префиксом `Источник: ` (если атрибуции ещё нет — §3a.1 требует атрибуцию в тексте claim'а); `status ← active`; `sources ← [<url страницы>]`.
3. **🔻 Осознанное ослабление контракта: `confidence` опционально для knowledge-claims.** §4 определяет `confidence` как **«оценку движка»** — у внешних утверждений источника на момент ингеста её нет, а единый дефолт был бы **выдуманными данными** (нарушение fidelity [ADR-0020]). Поэтому ингест `confidence` **не проставляет**; движок может добавить позже, когда оценит/кросс-сверит claim. `rules.md §3a.1/§4` правится, чтобы это зафиксировать.
4. **`status:` пакета → словарь контракта.** Пакет прошёл гейт качества (`jobsearch-kb-reviewer` PASS, ноль блокеров) → все ингестируемые страницы получают `status: active` (живой, одноисточниковый, ещё не owner-verified). `verified` НЕ выставляем: он требует ≥2 независимых источников **или** явного подтверждения владельца (§4) — это наступит при 2-м источнике (Фаза 3) или вручную.
5. **Транскрипт → `raw/` через fail-closed sanitizer.** Сырой источник (`yt-transcribe/transcript.txt`) прогоняется через [`failClosedSanitize`](../../src/ingest/sanitizer.ts); при `SanitizerError` запись отменяется целиком и watermark НЕ двигается. Результат — иммутабельный `raw/knowledge/job-search/nazarov-2024.md` (новый подкаталог); на него ссылается `url:` всех страниц. Полные детали источника в прозу `knowledge/` не копируются (§3a.4 «Провенанс»).
6. **Мирроринг дерева (идемпотентность по версии).** `knowledge/job-search/` приватного репо — **зеркало** `knowledge/job-search/**` пакета (трансформированное): страницы перезаписываются, а `.md`-файлы цели, которых нет в пакете, **удаляются** (так уходит seed-orphan `resume-ats.md`, у которого в пакете нет slug — топик разбит на `part-2-resume/*`). Идемпотентность — на уровне **версии пакета**: watermark (`raw/.watermarks/`, ключ `{package, version}`); та же версия → no-op, без дублей и без перезаписи `last_updated`. `advance`+`save` — только после успешных записей.

`docs/`, `.claude/` пакета **не** копируются (материалы разработки, MANIFEST §«Правила ингеста»). Граница public/private ([ADR-0003](0003-two-repos-public-private.md)) сохраняется структурно: дерево `knowledge/` существует только в приватном репо; `paid-thirdparty`-источник никогда не публикуется; `lint:public` — бэкстоп.

**Альтернатива (отклонена).** *Полностью ослабить контракт — принимать обе формы (`id/ref/source_date` и bare-string claims).* Дала бы чистую копию и портативность пакета as-is, но навсегда оставила бы движку **две формы** одного контракта (слабее маршрутизация/релевантность, сложнее проверки). Трансформация-при-ингесте держит **один канонический формат** в живой вике; пакет остаётся портативным в своём репо, а реконсиляция форм — единичная, детерминированная и идемпотентная (по версии). Ослабляем контракт **точечно** — только там, где альтернатива = выдумывать данные (`confidence`).

**Следствия.** Появляется `src/ingest/knowledge-package.ts` (оркестратор копия+трансформ+sanitize+watermark, переиспользует `failClosedSanitize`/`Watermark`, по образцу `telegram-export.ts`) + `knowledge-package.test.ts` (fail-closed sanitizer, идемпотентность по версии, корректность трансформа sources/claims/status, мирроринг-удаление orphan). `compiler/rules.md §3a.1/§4` уточняется: `confidence` — опционально для knowledge-claims; внешний пакет ингестируется трансформацией формы (не verbatim). Ре-релиз пакета (новая версия) ре-ингестируется тем же путём. Появление 2-го источника (Фаза 3) пойдёт по §3a.4 (обновление топиковых страниц + флаги расхождений), переиспользуя стабильные `claim_<4hex>`-id из п.2.
