# ADR-0017 — Telegram читает и продолжает сессии Claude Code (local-first, отдельная полоса, allowlist)

> Кристаллизовано в рабочем диалоге `/personal-llm-wiki` (2026-06-10). Расширяет интерфейс «Второго мозга» новой возможностью: владелец из Telegram **читает и продолжает свои локальные сессии Claude Code** (`~/.claude/projects/`). Уточняет [ADR-0007](0007-engine-spawn-and-scheduler.md)/[ADR-0015](0015-capture-write-path-permission-posture.md), **ничего не superseded** — wiki-полоса (cwd = приватный контент-репо, `acceptEdits`, без shell) остаётся буквально как есть.

**Контекст.** Владелец захотел «давать команды Второму мозгу через телегу, чтобы вести работу» — конкретно: видеть и продолжать сессии Claude Code, в которых он работает на этом же Mac (включая ту, где идёт этот диалог). Сырьё уже лежит локально: `~/.claude/projects/<encoded-cwd>/` — пер-проектные транскрипты `*.jsonl` + (в свежем формате) `sessions-index.json` с метаданными (`sessionId`, `projectPath`, `firstPrompt`, `messageCount`, `fileMtime`). На машине ~17 проектных папок, ~600 МБ транскриптов; среди них и личные, и **корпоративные abcage** (SQL/токены/Tracker в tool-output'ах).

Развилка, всплывшая в диалоге, и её разрешение:
- **Ложная дихотомия «перетащить проекты в облако».** Движок «Второго мозга» — это **локальный** бинарь `claude` ([ADR-0008](0008-engine-claude-native.md)/[ADR-0009](0009-tos-safe-engine-access.md)): он читает и правит файлы **на диске, на месте**. Проекты **не загружаются и не хранятся** ни на каком сервере. «Облако» — это только сама модель: на каждый ход в Anthropic API уходит **контекст этого хода** (куски, которые модель прочитала, чтобы ответить) — ровно как при любом обычном использовании Claude Code. Отдельного шага «залить репозиторий» нет. Это снимает исходный страх владельца и делает решение **local-first by construction**.
- **Lethal trifecta (триггер [ADR-0007]).** Нельзя просто дать wiki-движку `--add-dir ~/.claude/projects` + shell: это слило бы в один процесс приватную вику + чувствительный транскрипт + действие. Поэтому чтение/продолжение сессий — **отдельная полоса**, НЕ несущая контекст приватной вики и НЕ имеющая персоны «Второго мозга».

**Решение.**

1. **Local-first, граница «в сеть» явная.** Чтение сессий — это парсинг **локальных** файлов (`sessions-index.json` → fallback на head/tail `*.jsonl`); продолжение — локальный `claude -p --resume <session-id>` в cwd проекта сессии, правки идут в **локальные** файлы этого репозитория. Единственное, что покидает Mac: (а) текст команд/ответов через Telegram и (б) контекст хода для инференса в Anthropic API — **тот же объём**, что и при обычной работе в этой сессии напрямую.

2. **Отдельная полоса «сессии», изолированная от wiki-полосы.** Новый модуль моста `src/bridge/sessions.ts` + команды `/sessions [запрос]` · `/session <id>` · `/resume <id> <сообщение>`. Полоса продолжения строит **свой** движок (`buildEngineFromEnv(env, { repoPath: <projectPath> })`) с cwd = проект сессии и **без** `--append-system-prompt` персоны ([ADR-0016](0016-bot-persona-configurable-system-prompt.md)): это код-сессия, а не голос «Второго мозга», и приватная вика в неё не подмешивается. Так «приватные данные вики» и «shell/правка кода» **никогда не сосуществуют** в одном процессе — рассуждение lethal-trifecta [ADR-0007] сохранено. Wiki-полоса (`ClaudeEngine` на `WIKI_REPO_PATH`, постура `acceptEdits`/без shell из `.claude/settings.json` контент-репо) не тронута.

3. **Чтение — без облака; маскирование на отдачу.** `/sessions` и `/session` парсят файлы локально и **не** зовут движок. Всё, что уходит в Telegram (firstPrompt, хвост диалога), проходит `sanitizeText()` ([ADR-0011](0011-relevance-sensitivity-filter.md)) — маскирует секреты/PII, но **не блокирует** (read не должен падать на сессии, где когда-то светился токен).

4. **Продолжение — маскируем инъекцию fail-closed.** Сообщение владельца для `/resume` проходит `failClosedSanitize()` **до** движка (как реактивный ход, [ADR-0015](0015-capture-write-path-permission-posture.md) §2): сбой маскера → ход отменён. Постура правок полосы продолжения = **собственный `.claude/settings.json` целевого проекта** (владелец и так работает в этих репозиториях локально); мост не создаёт settings в чужих репозиториях (scope-дисциплина).

5. **Охват — `SESSIONS_ALLOWLIST` (deny-by-default).** Бот трогает только сессии, чей `projectPath` лежит под allowlist-префиксом. Пустой allowlist ⇒ фича выключена. В **публичном** фреймворке дефолт — пусто/выключено (`SESSIONS_ENABLED=0`), реальные пути — только в приватном `.env` (граница двух репо, [ADR-0003](0003-two-repos-public-private.md)). Владелец выбрал охват = **все проекты под `~/PhpstormProjects`** (включая рабочие abcage), осознанно приняв, что контекст рабочей сессии на `/resume` уходит в API — **наравне** с обычным локальным использованием Claude Code на тех же репозиториях.

6. **Owner-only без изменений.** Слой №2 ([ADR-0009](0009-tos-safe-engine-access.md), allow-list `TELEGRAM_OWNER_CHAT_ID`) и spawn-fresh + `--resume` ([ADR-0007](0007-engine-spawn-and-scheduler.md)/[ADR-0008](0008-engine-claude-native.md)) переиспользуются как есть — новых движковых примитивов нет.

**Альтернативы (отвергнуты).**
- **Ингест транскриптов в `wiki/`.** ~600 МБ чувствительного verbatim ломает контент-модель ([ADR-0010](0010-wiki-content-model.md): код-сессии → выжимка, не verbatim) и фильтр ([ADR-0011]); сессии — оперативный лог, а не стабильное знание. Отвергнуто. (Полезные итоги сессии по-прежнему можно осознанно записать в вику как accomplishment/capability — это ручной capture, не авто-ингест.)
- **`--add-dir ~/.claude/projects` + Bash у wiki-движка.** Lethal trifecta в одном процессе ([ADR-0007]). Отвергнуто в пользу отдельной полосы (Решение 2).
- **Облачная синхронизация/выгрузка проектов.** Не нужна и не существует в дизайне: движок локальный, файлы остаются на Mac (Решение 1). Отвергнуто как мнимая необходимость.

**Следствия и явное принятие риска.**
- **`/resume` корпоративной сессии шлёт её контекст в Anthropic API на инференс этого хода.** Это **принятый риск владельца**: эквивалентно тому, как он сам открывает ту же сессию в Claude Code. `failClosedSanitize` маскирует **новое инъектируемое сообщение**, но НЕ исторический транскрипт сессии — он и есть тот контекст, который модель легитимно подхватывает, чтобы продолжить (как в обычном ходе владельца). Границу того, что бот вообще готов трогать, держит `SESSIONS_ALLOWLIST`.
- **Триггер из Telegram = ход может пройти, пока владелец не смотрит в терминал.** Митигация: owner-only allow-list; постура правок — собственный `.claude/settings.json` проекта; продолжение требует явного `/resume <id>` (не авто).
- **Новая обязанность моста:** локальный индекс сессий (`sessions-index.json` + fallback на `*.jsonl`), дедуп по `sessionId`, сортировка по mtime; деградирует мягко (папка без индекса и без транскрипта → просто не появляется в списке).
- **Чтобы полоса продолжения могла править/гонять инструменты в проекте — у проекта должен быть свой `.claude/settings.json`** (у большинства dev-репозиториев он есть, либо владелец добавит). Мост его не навязывает.
- **Публичный фреймворк** демонстрирует фичу выключенной по умолчанию + `.env.example`-документацией; приватные пути и активация — в приватном репо ([ADR-0003]).
