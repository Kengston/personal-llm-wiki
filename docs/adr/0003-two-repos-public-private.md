# ADR-0003 — Два репозитория: публичный фреймворк vs приватный контент

**Контекст.** Проект должен быть и **портфолио** (открытый код на GitHub), и хранилищем сугубо **личных данных** (переписки, заметки обо мне). Эти две цели несовместимы в одном репо.

**Решение.** Разделяем на два git-репозитория (наследует `abcage-wiki` ADR-0004):
- **`personal-llm-wiki`** (публичный, GitHub `Kengston/...`) — фреймворк: `compiler/rules`, код `bridge`, скрипты `ingest`, `scheduler`, концепт-доки, **sanitized пример вики**, README/SETUP. Это то, что форкают и читают как портфолио.
- **`llm-wiki-content`** (приватный) — личные данные: `raw/`, `wiki/`, `reminders/`. Секреты — в `.env` (gitignored).

**Почему.** Жёсткая граница «код ≠ данные» — единственный надёжный способ гарантировать, что портфолио можно открыть, не утекая личным. Альтернатива (один приватный репо + ручная чистка перед публикацией) хрупкая: однажды забудешь почистить.

**Следствия.** Публичный репо получает **CI/линт на отсутствие PII и секретов** (см. [ADR-0006](0006-github-account-kengston.md) и `scheduler/lint`). Любой пример в публичном — синтетический/обезличенный. `sanitizer` ([ADR-0001](0001-engine-subscription-codex.md)-смежный инвариант) — первая линия обороны на write-path.
