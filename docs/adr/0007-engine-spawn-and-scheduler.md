# ADR-0007 — Паттерн спавна движка, формат reminders, принятие риска ToS/prompt-injection

> **Updated 2026-05-31 (engine-pivot).** ADR изначально писался под Codex как primary. Движок развёрнут на **Claude-native** ([ADR-0008](0008-engine-claude-native.md)) с ToS-безопасным доступом через официальный бинарь ([ADR-0009](0009-tos-safe-engine-access.md)). Engine-агностичные решения этого ADR (spawn-fresh-per-task, idempotent-sweep на launchd, формат `reminders.md`, минимизация blast radius prompt-injection) **остаются в силе** — они от движка не зависят. Engine-специфичные детали ниже переписаны на `claude -p` (исторический Codex-вариант сохранён как отложенный портируемый адаптер — [ADR-0008](0008-engine-claude-native.md)). Снят более не применимый риск «ToS подписочного Codex-backend»: см. перенос в [ADR-0009](0009-tos-safe-engine-access.md).

**Контекст.** Исследование 7 направлений (см. [docs/research/](../research/README.md)) выявило три конкретных решения, которые [ADR-0004](0004-telegram-bridge-reactive-proactive.md)/[ADR-0005](0005-host-v1-macbook-portable.md) и движковый [ADR-0008](0008-engine-claude-native.md) подразумевают, но не фиксируют явно. Их стоит закрепить, чтобы агенты-сборщики не разошлись в реализации.

**Решение.**

1. **Spawn-fresh-per-task, не резидентный демон.** И `bridge`, и `scheduler`/routine спавнят ОДИН короткоживущий процесс движка на задачу и дают ему выйти. Дефолт — `ClaudeEngine`: `claude -p "<prompt>" --output-format json` (парсим JSON → текст результата + `session_id`). Причина паттерна **engine-агностична**: короткоживущий процесс держит каждый ход **идемпотентным/crash-safe**, переживает ротацию токена/перезапуск и не копит состояние в долгоживущем демоне. Реактив — продолжение треда через `--resume <session_id>` (id хранится в SQLite `chat_id→session_id`) либо `--continue` для последней сессии; плановое/событийное — stateless, новый тред на запуск. Бинарь зовётся **только официальный** (ToS-safe, single-user — [ADR-0009](0009-tos-safe-engine-access.md)); OAuth-токен **никогда** не реюзается в стороннем HTTP-клиенте. Blast radius движка узкий: без generic «fetch any URL»/shell-инструмента сверх явно сконфигурированных MCP-капабилити (см. риск prompt-injection ниже).
   - *Отложенный `CodexEngine` (портируемый адаптер — [ADR-0008](0008-engine-claude-native.md)):* `codex exec` с теми же спавн-семантиками. Исторические Codex-нюансы, которые адаптер должен учесть: open-баг [#17041](https://github.com/openai/codex/issues/17041) (живая сессия падает на 401 при истечении токена и не подхватывает обновлённый `~/.codex/auth.json` — ровно поэтому spawn-fresh, а не резидент); **никогда** `--ephemeral` на resume-пути (баг [#15538](https://github.com/openai/codex/issues/15538): молча форкает новый thread); в `~/.codex/config.toml` — `forced_login_method="chatgpt"`, `approval_policy="never"`, `sandbox_mode="workspace-write"` с `writable_roots` на приватный репо и `network_access=false`. `GrokEngine` (тоже отложенный) — `grok -p --output-format json`; OpenClaw-бэкенд допустим **только** для Grok, не для Claude ([ADR-0009](0009-tos-safe-engine-access.md)).

2. **Планировщик — идемпотентный sweep, не cron.** Локальный v1 — **launchd** (LaunchAgent, не LaunchDaemon): при пробуждении из сна он запускает пропущенную задачу (cron скипает) и **коалесцирует** пропущенные интервалы в одно событие — поэтому job делает один «sweep» (спавнит `claude -p`, который читает все due-напоминания + дни рождения из вики → один Telegram-digest → помечает done), а не таймер на каждое напоминание. Дедуп через `status`/`last_fired`, чтобы коалесцированный двойной запуск не задвоил пуш — это и держит sweep идемпотентным независимо от того, кто его триггерит.
   - **Апгрейд — remote Claude routines (24/7).** launchd не сработает, пока Mac спит/выключен. Поскольку приватный контент живёт в GitHub-репо ([ADR-0006](0006-github-account-kengston.md)), плановые проходы (compile/ingest, утренний дайджест+напоминания, недельный lint, web-research, resurfacing идей) выносятся в **remote Claude routines**, которые гоняют `claude -p` над репозиторием **даже при спящем Mac**. Та же идемпотентность по `status`/`last_fired` делает локальный launchd и remote-routine взаимозаменяемыми и безопасными при наложении. См. [CONTEXT §4](../../CONTEXT.md), [ADR-0005](0005-host-v1-macbook-portable.md).

3. **Формат reminders (v1).** Автономный человекочитаемый файл `reminders/reminders.md` в приватном репо (НЕ CalDAV/Google Calendar — [CONTEXT §6 OQ-3](../../CONTEXT.md)) — append-only YAML-frontmatter-блоки. Поля: `id`, `title`, `kind` (oneoff|recurring|spaced), `due_at` (ISO 8601 с таймзоной), `rrule` (iCal RRULE, опц.), `nl_source` (исходная фраза, для аудита), `status`, `last_fired`, `created`; для spaced — `box`/`interval_days`/`ease` (лесенка Leitner `[1,3,7,16,35]`). **ISO-даты только**, относительные не персистим. iCal-RRULE-native формат держит поздний one-way-экспорт в `.ics` открытым без переделки.

**Почему отдельный ADR.** Это жёсткие, дорого-обратимые решения (паттерн процесса, формат данных reminders), на которые опираются `bridge`/`scheduler`/routine/`ingest`. Оставлять их только в research-докладе — риск дрейфа реализации.

**Принятие рисков (явное).**
- *ToS доступа к движку.* В Claude-native v1 движок зовётся **только официальным бинарём** под аккаунтом владельца (single-user) — это ToS-нативный, явно разрешённый режим; OAuth-токен не скрейпится в сторонний клиент. Полный разбор и стоимостная модель (Agent-SDK-кредит с 15.06.2026) — в [ADR-0009](0009-tos-safe-engine-access.md). *(Исходный риск «недокументированный Codex-backend `chatgpt.com/backend-api/codex/responses`» снят как неприменимый к Claude-native; он остаётся оговоркой для отложенного `CodexEngine`-адаптера.)*
- *Prompt-injection / lethal trifecta.* Система объединяет приватные данные + недоверенный ингестированный контент + внешнюю коммуникацию. 100%-защиты от prompt-injection НЕ существует — это от движка не зависит. Митигация — минимизация blast radius: least-privilege-доступ движка к ФС (запись только в приватный репо), единственный исходящий канал = узкий Telegram-пуш владельцу (owner-only allow-list), **без** generic «fetch any URL»/shell-инструмента сверх явно сконфигурированных MCP-капабилити. См. [docs/research/privacy-security.md](../research/privacy-security.md).
- *Приватность движка.* Для Claude-native — настройки приватности аккаунта владельца (ручной шаг в [setup/SETUP.md](../../setup/SETUP.md): проверить, что данные не идут в обучение, согласно текущим Consumer Terms; crown-jewel-секреты в прозу вики не писать). Та же дисциплина переносится на любой отложенный адаптер (для потребительских ChatGPT/Grok-аккаунтов — отдельно выключить тренинг на контенте). См. [ADR-0009](0009-tos-safe-engine-access.md), [docs/research/privacy-security.md](../research/privacy-security.md).

**Следствия.** `bridge`/`scheduler`/routine строятся вокруг `run_engine(prompt, session_id|None)`-абстракции (шов портируемости — абстрактный `Engine` с дефолтом `ClaudeEngine`, [ADR-0008](0008-engine-claude-native.md)). Trip-wire памяти ([ADR-0002](0002-no-embedder-pure-karpathy.md)) уточнён research-докладом конкретным порогом ~50–100K токенов / ~100–200 источников. Маленькая SQLite для `chat→session` и (опц.) дат reminders не нарушает ADR-0002 (он запрещает embedder/вектора для ВИКИ, не любую БД).
