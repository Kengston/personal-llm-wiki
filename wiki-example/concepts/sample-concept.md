# Концепция: «плоский файл побеждает базу»

> ⚠️ **СИНТЕТИКА.** Выдуманная концепция «Ивана Примера» для демонстрации формата `concepts/` (усвоенная ментальная модель). Все факты, источники и id — фейковые.

Это **ментальная модель**, а не идея на потом и не проект ([ADR-0010](../../docs/adr/0010-wiki-content-model.md)): здесь то, что владелец **понял и теперь применяет**. Концепции отделяют *усвоенное* от *задуманного* ([ideas/](../ideas/sample-idea.md)) и *сделанного* ([projects/](../projects/sample-project.md)). Сердце вики «про развитие» — про принципы мышления, не про код.

## Суть модели

Начинать решение задачи хранения/поиска с **самого простого носителя** — плоских markdown-файлов в git + `grep` — и усложнять (индекс, БД, ML) только когда простой слой **измеримо** перестаёт справляться. «Сложность по запросу боли, а не по предчувствию».

- [принцип] плоские файлы + `grep` закрывают ~90% поиска без индекса (`claim_9f2a`)
- [принцип] индекс/БД — только при реальной боли, не превентивно (`claim_4d10`)
- [следствие] id = `slug+дата` даёт дедуп «из коробки» без схемы и миграций
- [граница] модель ломается на больших объёмах / сложных join'ах — тогда осознанный переход на индекс

## Откуда пришло

Выкристаллизовалось из синтетического диалога с ассистентом при разборе [CLI-органайзера заметок](../projects/sample-project.md): я ждал, что без БД поиск «развалится», а на практике `grep` поверх файлов закрыл потребность. Тот же приём — watermark/идемпотентность в ингесте самой вики и отказ от эмбеддера в пользу чтения индекса ([ADR-0002](../../docs/adr/0002-no-embedder-pure-karpathy.md)).

## Как применяю

- В [CLI-органайзере](../projects/sample-project.md) — хранилище осознанно оставил на файлах+`grep`, индекс не заводил.
- В траектории роста — это маркер «тяга к минимализму осознанная, а не от незнания» ([growth/sample-growth.md](../growth/sample-growth.md)).
- Проверяю на новых замыслах как фильтр преждевременной оптимизации (ср. идею [тег-автодополнения](../ideas/sample-idea-from-chat.md): «не ML, а частотный список»).
