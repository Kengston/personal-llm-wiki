# Идея: тег-автодополнение для CLI-органайзера

> ⚠️ **СИНТЕТИКА.** Выдуманная идея «Ивана Примера» для демонстрации формата idea-страницы, **извлечённой из LLM-чата**. Имена, даты, id и переписка — фейковые.
>
> Демонстрирует контент-модель [ADR-0010](../../docs/adr/0010-wiki-content-model.md): инкрементальный ингест **чатов со всеми LLM** → идеи/концепции, плюс правило сжатия кода (из обсуждения в вику ушла **идея и решение**, а не код-простыня). Источник — синтетическая выгрузка диалога `raw/llm-chats/claude-2025-01-22-cli-tool.md` (поле `sources`).

В ходе разбора [CLI-органайзера заметок](../projects/sample-project.md) с ассистентом всплыла идея на потом: при `note add --tag` предлагать **автодополнение из уже использованных тегов**, чтобы не плодить синонимы (`work` / `работа` / `job`).

## Суть (из диалога)

- [идея] автодополнение тегов из истории — меньше дублей-синонимов
- [решение-из-чата] не нужен ML — хватит частотного списка прошлых тегов + префиксного матча
- [набросок] на старте `note` подгружать уникальные теги из файлов → fuzzy-подсказка
- [риск] не переусложнять; сперва проверить, мешают ли дубли вообще на практике

## Контекст происхождения

Извлечено компилятором из синтетической выгрузки чата (ChatGPT/Claude/Grok — ингест едино, [ADR-0010](../../docs/adr/0010-wiki-content-model.md)). В вику поднята **выжимка идеи и ключевое решение** «обойтись частотным списком без ML»; полный диалог с кодом остался в `raw/` и не дублируется здесь (правило сжатия кода).

## Открытые вопросы

- Дубли-синонимы — реальная боль или преждевременная оптимизация? Понаблюдать на практике.
- Если делать — отдельной мини-фичей или дождаться следующей итерации органайзера?
